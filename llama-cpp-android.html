I ran a local LLM on Android without any SDK — just Termux and llama.cpp
By Bibhash — Independent Developer & Algorithmic Trader
June 2026  ·  Bhubaneswar, India
No React Native. No Expo. No npm. No build machine. Just a terminal emulator, a compiler, and an Android phone with 8 GB of RAM.

A recent freeCodeCamp article walked through running a local LLM on Android using QVAC and Expo. It is a solid approach — clean SDK, good abstractions, React Native frontend. But it assumes you have a development machine to run the Expo build pipeline.

I do not have that. My entire setup lives on a Motorola Edge 20 running Termux. No laptop, no desktop, no external tools. So when I wanted a local LLM running on my phone, I had to go one level deeper — compile llama.cpp directly on the device, from source, on ARM64.

This is that story. And a guide so you can do it too.

Why not just use QVAC or a similar SDK?
The constraint

SDKs like QVAC are excellent when you have a build environment. They abstract away the hard parts — native compilation, memory mapping, hardware targeting. But they require npm, Expo, and a prebuild step that produces native binaries for Android or iOS.

On a phone-only setup, that pipeline simply does not work. Expo prebuild needs Node.js tooling that expects x86. Cross-compilation on ARM64 Termux for React Native targets is practically unsupported.

The key insight: llama.cpp is a C++ project that compiles natively on aarch64. Termux ships a full clang toolchain. That means you can build and run llama.cpp directly on your phone — no cross-compilation, no SDK, no laptop.

Step 1 — Set up Termux
Environment

Open Termux and run the following to update packages and grant storage access:

pkg update && pkg upgrade -y
termux-setup-storage
pkg install -y clang cmake git wget python

This installs the full build toolchain — clang compiler, cmake build system, git for cloning, and Python for the server wrapper later.

Step 2 — Clone and compile llama.cpp
Build

Clone the llama.cpp repository and build it natively on your device:

git clone https://github.com/ggerganov/llama.cpp
cd llama.cpp
mkdir build && cd build
cmake .. -DLLAMA_NATIVE=ON
make -j$(nproc)

The -DLLAMA_NATIVE=ON flag tells cmake to optimise for your exact CPU — important on ARM64 to get the best performance. The -j$(nproc) flag uses all available cores to speed up compilation.

This will take 10–20 minutes on a phone CPU. That is normal. Let it run.

If compilation fails: Try make -j2 instead to reduce memory pressure. Some phones with 4 GB RAM struggle with parallel compilation.
Step 3 — Download a model
Model

You need a GGUF format quantized model. The Q4_K_M quantization gives the best balance of quality and RAM usage. For a phone with 8 GB RAM, a 4B parameter model works well.

Good starting models from Hugging Face:

Qwen3 4B Q4_K_M (~2.6 GB)
Excellent reasoning, multilingual, fast on ARM64. This is what runs in my daily setup. Good for chat, code, and general Q&A.
Gemma 3 2B Q4_K_M (~1.5 GB)
Lighter, faster, good for 4 GB RAM devices. Less capable than Qwen3 4B but very responsive.
Phi-3.5 Mini Q4_K_M (~2.2 GB)
Microsoft's small model, surprisingly capable for its size. Good at instruction following.

Download directly in Termux:

# Create a models directory
mkdir -p ~/models
cd ~/models

# Download Qwen3 4B (replace URL with your chosen model's direct link)
wget -c "https://huggingface.co/<model-path>/model.Q4_K_M.gguf" -O qwen3-4b.gguf

The -c flag enables resume — if your download is interrupted, run the same command again and it picks up where it stopped.

Step 4 — Run the model as a local server
Inference server

llama.cpp ships a built-in HTTP server called llama-server. This exposes an OpenAI-compatible API on localhost, meaning any app or script that talks to the OpenAI API format works with it instantly.

~/llama.cpp/build/bin/llama-server \
  -m ~/models/qwen3-4b.gguf \
  --host 0.0.0.0 \
  --port 8080 \
  -c 2048 \
  -t 4 \
  --n-gpu-layers 0

Flag breakdown:

-mPath to your GGUF model file
--port 8080Port to serve the API on
-c 2048Context window size — reduce to 1024 if RAM is tight
-t 4Number of CPU threads — match your phone's core count
--n-gpu-layers 0Keep at 0 — phone GPUs are not supported by llama.cpp

Once running, test it from a second Termux session:

curl http://localhost:8080/v1/chat/completions \
  -H "Content-Type: application/json" \
  -d '{
    "model": "local",
    "messages": [{"role":"user","content":"Hello, are you running locally?"}]
  }'

You should see a JSON response with the model's reply — entirely generated on your phone's CPU, no internet, no API key.

Step 5 — Build a Python middleware layer
Integration

A raw llama-server is useful, but wrapping it in a Flask middleware lets you route between the local model and cloud APIs, add logging, and serve a web UI — all from one process.

pip install flask requests

# middleware.py
from flask import Flask, request, jsonify
import requests

app = Flask(__name__)
LOCAL_URL = "http://localhost:8080/v1/chat/completions"

@app.route("/chat", methods=["POST"])
def chat():
    data = request.json
    r = requests.post(LOCAL_URL, json=data, timeout=60)
    return jsonify(r.json())

if __name__ == "__main__":
    app.run(host="0.0.0.0", port=5000)
Tip: Use nohup python middleware.py & to keep the middleware running in the background after closing the Termux session. Add it to your startup script so everything launches automatically on boot.
How this compares to the QVAC + Expo approach
Comparison
QVAC + Expo (freeCodeCamp)
  • Needs npm + Node.js dev machine
  • React Native prebuild required
  • Clean SDK with loadModel() API
  • Works on iOS and Android
  • Great for app developers
  • Cannot run on Termux alone
llama.cpp + Termux (this guide)
  • Compiles natively on the phone
  • Zero external build tools needed
  • OpenAI-compatible API built in
  • Android only (ARM64)
  • Great for developers + hackers
  • Runs entirely on Termux

Neither approach is better — they solve different problems. QVAC is the right choice if you are building a mobile app to ship to users. The Termux + llama.cpp path is the right choice if you are building personal infrastructure on a phone you already own.

Six to ten tokens per second feels slow compared to a cloud API. In practice, for most tasks — summarising text, answering questions, writing code snippets — it is fast enough. The key benefit is not speed. It is privacy, zero cost, and zero internet dependency.

What I built on top of this
Real-world usage

The local llama-server is one component in a larger system. On top of it I built a Flask middleware that routes queries to the local model for offline or private tasks, and to cloud APIs (Gemini, Claude, DeepSeek) for heavier reasoning. A single HTML file serves as the unified chat UI connecting to all backends.

I also run a live algorithmic trading scanner, a Telegram bot assistant, and CA Foundation exam prep tools — all from the same phone. The local LLM handles tasks that should not leave the device. Everything else routes to the appropriate cloud API based on the task type.

Common issues and fixes
Troubleshooting
cmake: command not found
Run: pkg install cmake. Make sure you ran pkg update first.
Compilation killed / out of memory
Use make -j2 instead of make -j$(nproc) to reduce peak RAM usage during compilation.
Model loads but no output / gibberish
Check that your model is in GGUF format. GGML format (older) is not supported by recent llama.cpp builds.
Server stops when Termux session closes
Use nohup prefix or run inside a tmux session: pkg install tmux then tmux new -s llm.
Very slow inference (<3 tok/s)
Lower the context size with -c 512 and reduce threads to -t 2. Also try a smaller model like Gemma 2B.

Running a local LLM on Android without an SDK is not the easiest path. But it works, it costs nothing ongoing, it runs entirely offline, and it builds understanding that using an SDK cannot give you. When you compile llama.cpp on your phone, you understand exactly what is happening between your prompt and the model's response.

The best infrastructure is the infrastructure you understand completely. When it breaks — and it will break — you can fix it, because you built it yourself, from the ground up, on a phone.
