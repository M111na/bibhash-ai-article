AI Agents Android Termux Multi-Agent
Running two AI agents on one Android phone — how I built a multi-agent system in Termux
By Bibhash — Independent Developer & Algorithmic Trader
June 2026  ·  Bhubaneswar, India
Most multi-agent tutorials assume you have a server farm. I have a phone. Here is how I run two independent AI agents simultaneously — each with its own purpose, memory, and API backend — on a single Android device.

The phrase "multi-agent AI system" tends to conjure images of cloud infrastructure, orchestration frameworks, and GPU clusters. AutoGen, CrewAI, LangGraph — these tools are powerful, but they are built for machines with real compute budgets.

My setup is different. Everything runs on a Motorola Edge 20 — 8 GB RAM, ARM64, Termux. And on that single device, two separate AI agents run simultaneously, each handling a completely different domain. One is a general-purpose assistant. The other is a specialised trading and research agent. They never talk to each other. They do not need to. They just run.

This is how I built it, and how you can too.

The two agents
Architecture overview
Bibhash AI — General assistant
A Flask middleware on port 5000 routing queries to multiple backends: Gemini, Claude, DeepSeek via cloud APIs, and a locally running Qwen3 4B model via llama.cpp on port 8080. Handles general chat, code help, document analysis, and offline tasks. The local model handles anything that should not leave the device.
Aladdin — Trading and research agent
A standalone Python bot running independently on DeepSeek. Handles market research queries, trade analysis, and strategy reasoning. Runs as a separate process with its own context and memory. Completely isolated from Bibhash AI — different purpose, different backend, different process.
Key design principle: These agents are not orchestrated — they do not pass tasks to each other. They are independent processes serving different roles. This is intentional. Isolation means one agent crashing or being slow does not affect the other.
Why two agents instead of one?
Design rationale

The natural question is: why not just use one agent for everything and switch the system prompt?

A few reasons shaped this decision:

Context separation
A trading research session needs clean context — no noise from earlier general chat. Aladdin starts every session focused on markets. Bibhash AI handles everything else. Mixing them pollutes both.
Different backends, different strengths
DeepSeek handles financial reasoning and data analysis particularly well. Gemini handles long documents and multimodal tasks better. Having two agents lets each use the backend it is best suited for.
Process isolation
If the trading agent hits an API rate limit or crashes, the general assistant keeps running. They are separate OS processes — one dying does not kill the other.
Independent memory
Each agent maintains its own conversation history and state. Aladdin can remember that you are watching a particular stock setup without that context bleeding into a completely unrelated coding question on Bibhash AI.

The local LLM uses about 3.2 GB of RAM. Flask middleware and the Aladdin bot together use under 400 MB. Android's own overhead takes another ~1.5 GB. On 8 GB total, everything fits with room to spare.

On 4 GB RAM devices: Skip the local llama.cpp model and route everything to cloud APIs. The Flask middleware and Aladdin bot together use under 400 MB — they will run fine. You lose the offline/private model but keep the dual-agent architecture.
Building the general assistant — Bibhash AI
Agent 1

The general assistant is a Flask server that routes incoming requests to the right backend based on a mode parameter. Here is a simplified version of the core routing logic:

from flask import Flask, request, jsonify
import requests, os

app = Flask(__name__)

BACKENDS = {
  "gemini": "https://generativelanguage.googleapis.com/...",
  "deepseek": "https://api.deepseek.com/v1/chat/completions",
  "claude": "https://api.anthropic.com/v1/messages",
  "local": "http://localhost:8080/v1/chat/completions"
}

@app.route("/chat", methods=["POST"])
def chat():
  data = request.json
  mode = data.get("mode", "gemini")
  url = BACKENDS.get(mode, BACKENDS["gemini"])
  r = requests.post(url, json=data, timeout=60)
  return jsonify(r.json())

if __name__ == "__main__":
  app.run(host="0.0.0.0", port=5000)

The UI is a single HTML file that sends requests to this Flask server. Switching between models is just changing the mode value in the request — no restarts, no config changes.

Building the trading agent — Aladdin
Agent 2

Aladdin is a Telegram-native bot. Instead of a web UI, all interaction happens through Telegram messages. This is intentional — on a phone, Telegram is already open. No need to switch apps or open a browser.

import os, requests, time

TG_TOKEN = os.environ["TG_TOKEN"]
DS_KEY = os.environ["DEEPSEEK_KEY"]
CHAT_ID = os.environ["TG_CHAT_ID"]
offset = 0

SYSTEM = ("You are Aladdin, a professional trading and market "
          "research assistant. Be concise and data-focused.")

def ask_deepseek(user_msg):
  r = requests.post("https://api.deepseek.com/v1/chat/completions",
    headers={"Authorization": f"Bearer {DS_KEY}"},
    json={"model": "deepseek-chat",
         "messages": [{"role":"system","content":SYSTEM},
                    {"role":"user","content":user_msg}]},
    timeout=30)
  return r.json()["choices"][0]["message"]["content"]

def poll():
  global offset
  while True:
    r = requests.get(f"https://api.telegram.org/bot{TG_TOKEN}/getUpdates",
      params={"offset":offset,"timeout":20}, timeout=25)
    for u in r.json().get("result",[]):
      offset = u["update_id"] + 1
      msg = u.get("message",{}).get("text","")
      if msg:
        reply = ask_deepseek(msg)
        requests.post(f"https://api.telegram.org/bot{TG_TOKEN}/sendMessage",
          json={"chat_id":CHAT_ID,"text":reply})

poll()

Aladdin runs as a long-polling Telegram bot. No webhook, no server exposure — just an outbound poll every 20 seconds. Zero infrastructure requirements beyond the phone itself.

The startup script — launching everything at once
Automation

Both agents launch from a single bash script. Opening Termux and running one command brings the entire system online:

#!/data/data/com.termux/files/usr/bin/bash
# start_ai.sh — launch all agents

# Check if already running — avoid duplicate processes
if ! pgrep -f "llama-server" > /dev/null; then
  nohup ~/llama.cpp/build/bin/llama-server \
    -m ~/models/qwen3-4b.gguf \
    --port 8080 -c 2048 -t 4 \
    --n-gpu-layers 0 > ~/logs/llm.log 2>&1 &
  echo "LLM started on :8080"
else
  echo "LLM already running"
fi

if ! pgrep -f "middleware.py" > /dev/null; then
  nohup python ~/middleware.py > ~/logs/middleware.log 2>&1 &
  echo "Bibhash AI started on :5000"
else
  echo "Bibhash AI already running"
fi

if ! pgrep -f "aladdin_bot.py" > /dev/null; then
  nohup python ~/aladdin_bot.py > ~/logs/aladdin.log 2>&1 &
  echo "Aladdin started"
else
  echo "Aladdin already running"
fi

echo "=== All agents online ==="
The pgrep check is critical. Without it, opening a new Termux session and re-running the script would spawn duplicate processes — two copies of Aladdin both responding to every Telegram message, two Flask servers fighting over port 5000. Always check before launching.
Monitoring both agents
Health checks

Keeping track of two background processes on a phone requires a lightweight status check. Add this to your startup script output or run it any time:

# Quick status check
echo "=== Agent Status ==="
pgrep -f "llama-server" > /dev/null && echo "LLM : [OK]" || echo "LLM : [DOWN]"
pgrep -f "middleware.py" > /dev/null && echo "BibhashAI: [OK]" || echo "BibhashAI: [DOWN]"
pgrep -f "aladdin_bot" > /dev/null && echo "Aladdin : [OK]" || echo "Aladdin : [DOWN]"
echo "RAM used: $(free -m | awk '/Mem/{print $3}') MB"

This is exactly what you see in the Termux startup banner — a quick confirmation that everything is running before you start your day.

What makes this different from typical multi-agent frameworks
Comparison
This approach (Termux)
  • Plain Python processes
  • No orchestration framework
  • Agents are fully isolated
  • Runs on 8 GB phone RAM
  • Zero cloud infrastructure
  • Single bash script to launch
  • Survives phone restarts
AutoGen / CrewAI / LangGraph
  • Framework-managed agents
  • Built-in orchestration
  • Agents can pass tasks
  • Needs 16 GB+ RAM typically
  • Often requires cloud deploy
  • Complex setup and config
  • Heavier dependencies

The framework approach is better for complex agent-to-agent pipelines where one agent's output feeds another. For two independent specialists running in parallel — serving different purposes, different users, different contexts — plain processes are simpler, lighter, and more reliable.

Common problems and fixes
Troubleshooting
Aladdin not responding after phone sleep
Android aggressively kills background processes to save battery. Install Termux:Boot from F-Droid and place your startup script in ~/.termux/boot/ to auto-relaunch on wake. Also disable battery optimisation for Termux in Android settings.
Port 5000 already in use on restart
The pgrep check handles this — but if a zombie process holds the port, run: kill $(lsof -ti:5000) before restarting.
Both agents slow when LLM is running inference
The local LLM uses all CPU threads during inference. Set -t 2 instead of -t 4 in llama-server to leave cores free for other processes. Response will be slower but the system stays responsive.
Logs growing too large
Add log rotation: nohup python middleware.py 2>&1 | tail -c 1M > ~/logs/middleware.log & or simply truncate logs weekly with a cron job.

Two AI agents, one phone, 8 GB of RAM. No cloud infrastructure. No orchestration framework. No monthly server bill. The system has been running daily since it was built — handling general questions on one agent while monitoring markets and answering research queries on the other.

The real insight is not the technology. It is the design decision: keep agents independent, keep them simple, and let the operating system do the scheduling. That approach scales down to a phone just as well as it scales up to a server.

Every system that runs reliably does so because someone resisted the urge to make it more complex than it needed to be. Two processes, two purposes, one phone. That is enough.
